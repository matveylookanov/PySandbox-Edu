# Тестирование системы «PySandbox Edu»

## 1. Цели тестирования

- Проверка соответствия реализованной системы требованиям (СТПО, ТЗ).
- Обнаружение функциональных и нефункциональных дефектов.
- Оценка безопасности и изоляции выполнения пользовательского кода.

## 2. Объект тестирования

- Backend (FastAPI + Docker SDK).
- Frontend (HTML/CSS/JS).
- Docker-окружение (backend, frontend, executor).

## 3. Общий тест-план

1. **Установочное тестирование**:
   - проверка наличия всех файлов и структуры проекта;
   - проверка успешного запуска `docker-compose up --build`.
2. **Функциональное тестирование backend**:
   - тесты эндпоинтов `/`, `/health`, `/execute`.
3. **Функциональное тестирование frontend**:
   - ввод кода, запуск, отображение результатов.
4. **Тестирование ограничений и безопасности**:
   - проверка лимита длины кода;
   - проверка таймаута;
   - попытки доступа к файловой системе и сети из контейнера.
5. **Тестирование отказоустойчивости**:
   - поведение при недоступном Docker;
   - поведение при отсутствии образа `pysandbox-executor`.

## 4. Примеры тест-кейсов

### TC-001: Успешный запуск простого кода

- **Цель**: проверить базовый сценарий выполнения.
- **Предусловия**: запущен `docker-compose up --build`.
- **Шаги**:
  1. Открыть `http://localhost:3000`.
  2. Ввести в поле кода: `print("Hello, PySandbox")`.
  3. Нажать «Запустить код».
- **Ожидаемый результат**:
  - `stdout`: `Hello, PySandbox` (с переводом строки);
  - `stderr`: пусто;
  - код завершения: `0`;
  - время выполнения указано в миллисекундах;
  - статус в интерфейсе: «Готово».

### TC-002: Ошибка синтаксиса Python

- **Шаги**:
  1. Ввести некорректный код, например: `if True print("oops")`.
  2. Нажать «Запустить код».
- **Ожидаемый результат**:
  - `stdout`: пусто;
  - `stderr`: сообщение об ошибке синтаксиса Python (Traceback);
  - код завершения: не 0 (например, 1);
  - система не падает, интерфейс отображает результат.

### TC-003: Превышение лимита длины кода

- **Шаги**:
  1. Ввести строку более 10 000 символов (можно сгенерировать).
  2. Нажать «Запустить код».
- **Ожидаемый результат**:
  - HTTP-ответ 400;
  - сообщение об ошибке с указанием максимальной длины;
  - контейнер для выполнения кода не создаётся.

### TC-004: Таймаут выполнения

- **Шаги**:
  1. Ввести код с бесконечным циклом, например:

     ```python
     while True:
         pass
     ```

  2. Нажать «Запустить код».
- **Ожидаемый результат**:
  - выполнение прерывается по таймауту (например, через 10 секунд);
  - контейнер принудительно завершается;
  - пользователь получает сообщение об ошибке таймаута.

### TC-005: Попытка доступа к сети

- **Шаги**:

  ```python
  import socket
  socket.gethostbyname("google.com")
  ```

- **Ожидаемый результат**:
  - код не должен иметь доступа к внешней сети (ошибка подключения или невозможность установить соединение);
  - backend и хост-система остаются работоспособными.

### TC-006: Попытка доступа к файловой системе хоста

- **Шаги**:

  ```python
  import os
  print(os.listdir("/"))
  ```

- **Ожидаемый результат**:
  - видна только файловая система контейнера, нет доступа к критичным директориям хоста;
  - отсутствует доступ к файлам проекта вне контейнера.

### TC-007: Недоступен Docker

- **Шаги**:
  1. Остановить Docker Engine (или запустить backend без монтирования `/var/run/docker.sock`).
  2. Вызвать `/health` и `/execute`.
- **Ожидаемый результат**:
  - `/health` возвращает статус `"degraded"` и `docker: false`;
  - `/execute` возвращает ошибку 500 с сообщением о недоступности Docker.

## 5. Чек-лист для приёмочного тестирования

- [ ] Структура проекта соответствует заданию.
- [ ] `docker-compose up --build` выполняется без фатальных ошибок.
- [ ] `GET /health` возвращает `"status": "ok"` при запущенном Docker.
- [ ] Простые программы выполняются корректно.
- [ ] Лимит длины кода работает.
- [ ] Таймаут срабатывает при бесконечном цикле.
- [ ] Попытки сетевого доступа из кода не приводят к доступу во внешнюю сеть.
- [ ] Логи запросов и ошибок сохраняются (консоль/файлы логирования по решению команды).

## 6. Баг-репорты

Баг-репорты оформляются по единому шаблону:

- ID/номер;
- заголовок;
- окружение (OS, версия Docker, версия образов);
- шаги воспроизведения;
- фактический результат;
- ожидаемый результат;
- вложения (логи, скриншоты).

Рекомендуется создавать баг-репорты в GitHub Issues.
